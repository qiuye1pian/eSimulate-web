# 新版前端重构设计

本文档描述 `eSimulate-web` 新版前端的重构目标、技术选型、架构设计和迁移边界。新版前端的核心目标是：功能与旧版保持一致，技术栈升级到现代 React 体系，并重新组织代码架构，避免继续继承旧版代码中的坏味道。

## 一、重构目标

新版前端不是重新定义产品，而是对旧版前端 `WebReactHuaBeiPower` 做等功能复刻和工程重建。

目标如下：

- 功能保持与旧版一致。
- 接口协议尽量沿用旧版后端接口，降低后端改造成本。
- 技术栈升级到现代、长期可维护的 React 生态。
- 页面结构、状态管理、接口层、图表、表单和轮询任务重新设计。
- 支持 Docker 化构建和部署。
- 第一阶段优先完成核心业务闭环，系统管理模块根据实际使用情况决定优先级。

不做的事情：

- 不逐行翻译旧代码。
- 不继续使用 React 16、webpack 4、Ant Design 3、React Router 3。
- 不把旧版中的历史示例页、无效注释、调试日志和明显 bug 原样迁移。

## 二、技术选型

建议技术栈：

```text
React 18
Vite
TypeScript
Ant Design 5
React Router 6
Axios
TanStack Query
Zustand
ECharts
Vitest
ESLint
Prettier
Docker + Nginx
```

选型理由：

- React 与旧版方向一致，迁移业务理解成本低。
- Vite 构建快，配置简单，适合新项目。
- TypeScript 能约束接口 DTO、表单数据和图表数据结构。
- Ant Design 5 适合管理后台、表格、表单、弹窗和布局。
- TanStack Query 适合列表查询、详情查询、上传后刷新、轮询任务。
- Zustand 适合认证信息、菜单、布局偏好等轻量全局状态。
- ECharts 与旧版图表能力一致，迁移曲线图、饼图、堆叠图成本低。
- Docker + Nginx 适合部署静态前端资源。

目录原则：

- `features` 放业务模块内部组件、hooks、类型和局部逻辑。
- `services` 统一管理后端接口。
- `components` 只放跨业务复用组件。
- `stores` 只放真正需要跨页面共享的轻量状态。
- DTO 和枚举集中放在 `types` 或业务模块内部的 `types.ts`。

## 四、路由设计

旧版使用 hash 路由，新版建议使用 React Router 6。是否继续使用 hash 模式取决于部署环境：

- 如果 Nginx 可以配置 history fallback，推荐 browser history。
- 如果部署环境不方便改 Nginx，保留 hash router 也可以。

第一阶段路由建议：

```text
/login

/simulation/scheme
/simulation/optimization

/environment/electric-load
/environment/thermal-load
/environment/grid-pricing
/environment/wind
/environment/water-flow
/environment/sunlight
/environment/temperature

/model/wind-power
/model/photovoltaic
/model/hydropower
/model/battery
/model/solar-thermal
/model/gas-boiler
/model/thermal-power-unit
/model/cogeneration
/model/pumped-storage
/model/thermal-storage

/system/users
/system/roles
```

旧版被注释或不确定的模块：

- `GasSupply` 燃气供应：旧路由中被注释，但代码和接口存在。新版先在设计上保留，实施时确认是否启用。
- `Simulation/Result` 仿真结果：旧路由中被注释，结果展示已集成在方案仿真页。新版不单独做结果页，除非后续确认需要。
- `system` 系统管理：旧路由配置没有挂载，但代码存在。新版先保留设计位置，优先级可后置。

## 五、功能模块设计

### 1. 登录与认证

旧版能力：

- 获取 RSA 公钥。
- 用户名密码登录。
- 密码前端 RSA 加密。
- token 存入 `sessionStorage`。
- 菜单数据写入 `sessionStorage`。
- 支持 `ticket`、`key`、iframe 模式。

新版设计：

- `auth` 模块封装登录、登出、公钥获取、用户信息、菜单获取。
- 使用 `auth-store` 管理 token、用户、菜单。
- token 可以继续存 `sessionStorage`，但菜单和用户信息优先通过接口恢复。
- 路由守卫放在统一的 `RequireAuth` 组件中。
- `ticket`、`key`、iframe 模式作为兼容能力保留，但不要污染主登录流程。

### 2. 环境与负荷数据

覆盖模块：

- 电负荷。
- 热负荷。
- 风力数据。
- 水流数据。
- 光照数据。
- 温度数据。
- 电网电价。
- 燃气供应待确认。

新版设计：

- CSV 类数据页面使用统一模板。
- 列表、分页、搜索、上传、下载、删除、图表预览统一封装。
- 电网电价作为表单型资源单独配置字段。
- 曲线图使用统一 `LineChart` 或 `CurveChart` 组件。

建议抽象：

```text
CsvSeriesResourcePage
ResourceList
CsvUploadModal
DownloadButton
CurvePreview
```

### 3. 模型配置

覆盖模块：

- 风电。
- 光伏。
- 小水电。
- 电储能。
- 太阳能集热。
- 燃气锅炉。
- 火电机组。
- 热电联产。
- 抽水蓄能。
- 热储能。

新版设计：

- 用配置驱动大部分模型页面。
- 不同模型声明自己的字段、单位、校验规则、接口、图表能力。
- 列表、选中项、表单编辑、新增、保存、删除、图形预览使用统一骨架。
- 小水电等带公式计算的模型保留专用扩展逻辑。

建议抽象：

```text
ModelConfigPage
ModelListPanel
ModelFormPanel
ModelCurvePreview
modelDefinitions
```

### 4. 方案仿真

旧版流程：

- 选择负荷。
- 选择模型。
- 选择环境。
- 配置各模型数量。
- 调用 `/api/simulate/simulator/do`。
- 展示指标、饼图、堆叠图。

新版设计：

- 将资源选择拆成独立组件。
- 将 DTO 组装逻辑放入纯函数，便于测试。
- 将仿真提交和结果状态放入 `useSimulation`。
- 指标卡片、饼图、堆叠图拆成独立展示组件。

关键 DTO：

```text
environmentDtoList
loadDtoList
modelDtoList
```

### 5. 寻优计算

旧版流程：

- 选择负荷、模型、环境。
- 配置粒子群算法参数和约束。
- 调用 `/api/optimize/pso/do` 创建任务。
- 每 3 秒查询 `/api/optimize/pso/getTaskState`。
- 完成后调用 `/api/optimize/pso/getResult`。
- 展示结果表格。
- 支持取消任务。

新版设计：

- 使用 `useOptimizationTask` 管理提交、轮询、完成、失败、取消。
- 轮询由 TanStack Query 或自定义 hook 控制，组件卸载自动停止。
- 参数表单独立成 `OptimizationConfigForm`。
- 结果表格单独成 `OptimizationResultTable`。
- 离开页面和刷新页面时取消任务的逻辑需要谨慎处理，避免误取消已完成任务。

关键 DTO：

```text
environmentDtoList
loadDtoList
modelDimensionDtoList
particleCount
maxIterations
inertiaWeightStart
inertiaWeightEnd
c1Start
c1End
c2Start
c2End
maxCurtailmentRate
minRenewableEnergyShare
```

### 6. 系统管理

旧版存在用户、角色、模块管理代码，但当前实际业务反馈没有资源管理功能。

新版策略：

- 设计上保留 `system` 模块。
- 第一阶段是否实现需要确认业务是否仍使用。
- 如果实现，优先复刻用户和角色两个页面。
- 权限按钮能力可以先保留接口设计，后续逐步接入。

## 六、接口层设计

旧版接口封装问题：

- callback 风格，不利于组合和错误处理。
- 下载接口靠第三个参数切换 blob，不直观。
- 生产地址写死。
- token 没有清晰注入请求头。
- 成功/失败结构没有类型约束。

新版接口层：

```text
services/request.ts
services/auth.ts
services/environment.ts
services/model-config.ts
services/simulation.ts
```

设计原则：

- 所有接口返回 Promise。
- 统一响应类型，例如 `ApiResponse<T>`。
- 统一错误处理和登录过期处理。
- 下载接口单独封装。
- API base URL 从环境变量读取。
- 兼容旧版后端 POST 风格。

环境变量示例：

```text
VITE_API_BASE_URL=http://localhost:8000
VITE_API_PREFIX=/api
```

## 七、状态管理设计

全局状态只保留必要内容：

- 认证状态。
- 当前用户。
- 菜单树。
- 布局偏好，如侧边栏折叠。

服务端数据交给 TanStack Query：

- 列表查询。
- 详情查询。
- 上传后刷新。
- 删除后刷新。
- 仿真结果。
- 寻优任务轮询。

组件局部状态留在组件内：

- 弹窗显隐。
- 当前表单编辑状态。
- 当前选中行。
- 当前展开面板。

## 八、UI 与交互原则

新版以业务操作台为目标，设计风格应克制、清晰、稳定。

原则：

- 不做营销页，不做大面积装饰性首页。
- 主要界面优先服务数据录入、配置、比较和结果查看。
- 页面布局密度适中，避免旧版页面过长且信息混乱。
- 列表与表单分区明确。
- 图表区域尺寸稳定。
- 上传、下载、删除、保存、运行仿真、开始寻优等动作需要明确反馈。
- 表单校验和错误提示要前置，不等后端报错才发现明显问题。

## 九、Docker 部署设计

新版前端需要提供 Dockerfile。

建议多阶段构建：

1. Node 阶段安装依赖并执行 `npm run build`。
2. Nginx 阶段托管 Vite 生成的 `dist` 目录。

需要提供：

```text
Dockerfile
.dockerignore
nginx.conf
.env.example
```

Nginx 需要支持：

- 静态资源托管。
- 前端路由 fallback。
- 可选的 `/api` 反向代理。

是否在 Nginx 内代理后端，需要根据部署方式确认：

- 如果前后端同容器网络部署，Nginx 可以代理 `/api`。
- 如果由外层网关统一代理，前端镜像只托管静态资源。

## 十、迁移策略

推荐迁移顺序：

1. 创建 Vite + React + TypeScript 项目骨架。
2. 加入 Dockerfile、Nginx、环境变量和基础工程配置。
3. 实现登录、主布局、路由守卫和固定菜单。
4. 实现接口层和统一请求封装。
5. 实现环境/负荷 CSV 类资源页面模板。
6. 实现模型配置页面模板。
7. 复刻全部环境/负荷页面。
8. 复刻全部模型配置页面。
9. 实现方案仿真页面。
10. 实现寻优计算页面。
11. 根据需要实现系统管理模块。
12. 做接口联调、构建验证和 Docker 运行验证。

## 十一、验收标准

第一阶段完成时，应满足：

- 新版前端可以本地启动。
- 新版前端可以 Docker 构建。
- 登录流程可用。
- 旧版核心业务路由在新版中有对应页面。
- 环境/负荷数据支持列表、搜索、上传、下载、删除和图表预览。
- 模型配置支持列表、选择、编辑、新增、删除和必要图表预览。
- 方案仿真能提交旧版兼容 DTO 并展示结果。
- 寻优计算能创建任务、轮询状态、展示结果并取消任务。
- 关键接口类型、枚举和 DTO 有 TypeScript 类型定义。
- 不再出现直接修改 React state、超大页面堆逻辑、匿名 resize 监听无法清理等旧版问题。
