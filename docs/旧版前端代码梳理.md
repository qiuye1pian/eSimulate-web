# 旧版前端代码梳理

本文档用于记录旧版前端 `code/frontend/WebReactHuaBeiPower` 的项目结构、业务模块、接口关系和迁移风险，作为新版前端 `code/frontend/eSimulate-web` 复刻开发的上下文。

## 一、总体结论

旧版前端是一个 React 管理端项目，主要面向“县域多能互补一体化平台”的环境数据管理、模型配置、方案仿真和寻优计算。

当前技术栈较老：

- React 16
- React Router 3，使用 `hashHistory`
- Redux 3 / react-redux 5
- Ant Design 3
- Webpack 4
- Axios 0.19
- ECharts 5
- Jest + Enzyme

新版前端建议采用：

- React 18
- Vite
- TypeScript
- Ant Design 5
- React Router 6
- TanStack Query
- Zustand
- ECharts

部署要求：

- 新版前端需要提供 Dockerfile。
- 建议采用多阶段构建：Node 阶段执行依赖安装和 Vite 构建，Nginx 阶段托管静态产物。
- API 地址、后端代理和生产环境配置不应写死在代码中，应通过 `.env`、构建参数或 Nginx 配置注入。

## 三、已识别业务模块

### 1. 登录与主框架

相关文件：

- `app/pages/base/login.js`
- `app/pages/base/app.js`
- `app/pages/base/app/header.js`
- `app/pages/base/app/leftNav.js`
- `app/configs/common.js`
- `app/apis/common.js`

主要功能：

- 用户名密码登录。
- 登录前请求 RSA 公钥，前端使用 `jsencrypt` 加密密码。
- 登录成功后把 token、菜单、顶级菜单等信息写入 `sessionStorage`。
- 主框架包含顶部导航、左侧导航和内容区。
- 支持 `ticket` 登录、`key` 模式和 iframe 模式。

注意事项：

- 登录页实际菜单来自本地 mock 文件 `app/mocks/apis/base/menu`，而不是完全依赖后端菜单接口。
- `sessionStorage` 使用很多，菜单、token、导航状态、当前菜单 id 都依赖它。
- `app/configs/common.js` 中 `_fetchNav` 调用了 `nav(...)`，但当前文件没有导入 `nav`，而 `app/apis/common.js` 中实际导出的是 `menu`。这是一个明确的潜在 bug。

### 4. 方案仿真

相关文件：

- `app/pages/Simulation/scheme/index.js`
- `app/apis/Simulation.js`

路由：

```text
/Simulation/Scheme
```

接口：

```text
POST /api/simulate/simulator/do
```

页面结构：

- 折叠面板 1：负荷选择。
- 折叠面板 2：模型配置选择。
- 折叠面板 3：环境配置选择。
- 折叠面板 4：数量配置。
- 折叠面板 5：仿真结果展示。

提交参数由三类列表组成：

```text

```

返回结果主要用于展示：

- 可再生能源占比。
- 弃风弃光率。
- 碳排放总量。
- 年度总成本。
- 电负荷堆叠图。
- 热负荷堆叠图。

### 5. 寻优计算

相关文件：

- `app/pages/Simulation/seeking/index.js`
- `app/apis/Simulation.js`

路由：

```text
/Simulation/Seeking
```

接口：

```text
POST /api/optimize/pso/do
POST /api/optimize/pso/getTaskState
POST /api/optimize/pso/getResult
POST /api/optimize/pso/cancelTask
```

主要流程：

1. 用户选择负荷、模型、环境。
2. 用户配置粒子群算法参数和约束。
3. 调用 `optimizedo` 创建任务。
4. 后端返回 `taskId`。
5. 前端每 3 秒轮询任务状态。
6. 状态为 `COMPLETED` 后调用结果接口。
7. 用表格展示每个结果位置和值。

寻优参数包括：

- `particleCount`
- `maxIterations`
- `inertiaWeightStart`
- `inertiaWeightEnd`
- `c1Start`
- `c1End`
- `c2Start`
- `c2End`
- `maxCurtailmentRate`
- `minRenewableEnergyShare`

风险点：

- 当前页面超过 1400 行，状态、表单、轮询、结果表格全部混在一个类组件里。
- 多处直接修改 `this.state.formValues`，例如 `this.state.formValues.particleCount = e`。
- 已经有轮询清理逻辑，但需要在新版中统一封装为任务 hook，避免内存泄漏和重复轮询。

## 四、接口封装现状

旧版接口统一由 `app/configs/ajax.js` 的 `createApi` 和 `oftenFetchByPost` 创建。

特点：

- 默认请求方法是 POST。
- 默认 `withCredentials: true`。
- 下载接口通过第三个参数 `num` 控制 `responseType = 'blob'`。
- 请求成功后以 callback 形式返回，而不是 Promise/async。
- 通过解析 URL 上的 `key` 和 `sourceName` 拼接到接口地址。
- token 存在 `sessionStorage`，但目前没有实际加到请求体或请求头里。

新版建议：

- 使用 Axios 实例封装 `request`。
- 所有接口函数返回 Promise。
- 下载接口单独封装 `downloadBlob`。
- 接口层用 TypeScript 定义请求和响应类型。
- 使用 TanStack Query 管理列表、详情、轮询任务和缓存失效。

## 五、代码坏味道与潜在 bug

### 1. 类组件和装饰器依赖较重

大量页面使用：

```text
@Form.create(...)
class xxx extends Component
```

这依赖 Babel 装饰器和 Ant Design 3 的表单模式。新版应改为函数组件、hooks 和 Ant Design 5 的 `Form.useForm()`。

### 2. 页面文件过大

统计到的最大文件：

```text
Simulation/seeking/index.js                         1467 行
ModelConfiguration/Cogeneration/index.js            1024 行
ModelConfiguration/Hydropower/index.js              1022 行
ModelConfiguration/EnergyStorage/index.js            966 行
Simulation/scheme/index.js                           951 行
ModelConfiguration/Thermalpowerunit/index.js         931 行
```

这些页面同时承担列表、表单、图表、状态管理、接口请求和业务 DTO 组装，复刻时应该拆分。

### 3. 重复代码很多

环境数据页面、电负荷/热负荷页面、模型配置页面有大量相同结构：

- 分页状态。
- 搜索关键字。
- 当前选中行。
- 新增/编辑弹窗。
- 删除确认。
- 图表配置。
- 下载和上传。

新版应设计配置化页面模板，减少复制。

### 4. 直接修改 state

旧代码中多处存在：

```text
this.state.paging.page = e
this.state.formValues.xxx = e
this.state.selectedRowKeys = ...
```

这会绕过 React 状态更新机制，容易造成 UI 不刷新、状态错乱或难以复现的 bug。

### 5. 登录和菜单状态耦合过强

登录成功后把 token、菜单、左侧导航、顶级菜单写入 `sessionStorage`。主框架又直接读取这些字段。

新版建议：

- 将认证状态、用户信息、菜单树统一放在 auth store。
- 刷新页面时通过接口恢复用户和菜单。
- `sessionStorage` 只保留 token 或必要的临时状态。

### 6. 明确的命名/导入问题

`app/configs/common.js` 中 `_fetchNav` 调用了 `nav`，但没有导入。`app/apis/common.js` 导出的是 `menu`。

这段逻辑如果执行，会出现 `nav is not defined`。

### 7. ECharts resize 监听清理不可靠

部分组件使用匿名函数注册和移除：

```text
window.addEventListener('resize', () => ...)
window.removeEventListener('resize', () => ...)
```

这两个匿名函数不是同一个引用，卸载时无法正确移除监听。新版应封装统一的 `useECharts` 或图表组件。

### 8. 生产环境地址写死

`app/configs/config.js` 中生产地址写死为：

```text
http://47.113.180.169:80
```

新版应使用 `.env` 配置，例如：

```text
VITE_API_BASE_URL
```

### 9. 注释、console、历史代码较多

旧代码保留了大量注释代码、`console.log`、`TODO/FIXME` 和示例页面。新版复刻时不应逐行迁移这些历史包袱。

## 六、新版复刻建议

### 1. 先搭建业务骨架

建议先按业务模块建立路由：

```text
/login
/simulation/scheme
/simulation/optimization
/environment/electric-load
/environment/thermal-load
/environment/grid-pricing
/environment/wind
/environment/water-flow
/environment/sunlight
/environment/temperature
/model/wind-power
/model/photovoltaic
/model/hydropower
/model/battery
/model/solar-thermal
/model/gas-boiler
/model/thermal-power-unit
/model/cogeneration
/model/pumped-storage
/model/thermal-storage
```

### 2. 先抽象三类页面模板

建议优先抽象：

- `ResourceListPage`：列表、分页、搜索、选择、删除。
- `CsvSeriesPage`：CSV 上传、下载、曲线预览。
- `ModelConfigPage`：模型参数表单、保存、删除、图形预览。

仿真和寻优页面再基于这些资源选择组件组合。

### 3. 先复刻后优化交互

当前任务是复刻，因此建议第一阶段保持业务流程和接口参数兼容：

- 路由可现代化，但业务页面数量保持一致。
- 接口路径和 DTO 枚举保持一致。
- 表单字段名可内部重命名，但提交给后端的数据结构必须兼容。
- 明显 bug 和坏味道在新版中直接规避，不做“逐行翻译”。

### 4. 优先保留的核心流程

第一阶段建议优先复刻：

1. 登录。
2. 负荷/环境数据管理。
3. 模型配置。
4. 方案仿真。
5. 寻优计算。
6. 结果图表展示。

系统管理模块可根据实际是否仍在使用决定是否纳入第一阶段。

## 七、下一步建议

下一步可以开始写新版前端的设计文档和实施计划，建议输出：

```text
code/frontend/eSimulate-web/docs/新版前端重构设计.md
code/frontend/eSimulate-web/docs/新版前端实施计划.md
code/frontend/eSimulate-web/Dockerfile
```

在真正创建 Vite 项目前，应先确认：

- 是否完整复刻系统管理模块。
- 是否沿用旧版接口路径。
- 登录菜单是否继续使用后端菜单，还是先使用固定菜单。
- 新版 UI 是否只做现代化整理，还是重新设计信息架构。
- Docker 镜像是否只负责托管前端静态资源，还是还要内置 Nginx 反向代理到后端。
